Socket programming provides a way of communication between two nodes on a network. One socket(node) listens on a particular port at an IP address, while another socket reaches out to the other to form a connection. The server creates a listener socket while the client initiates a connection to the server.

Network communications can be initiated between any 2 programs, whether they are running on the same machine, on different machines on the same local area network, or on different machines across the Internet. When you work on your next assignment you may test with local connections or by running the client/server programs running on the two different Jupyter servers respectively.

In this lab, the goal is to experience the use of basic socket programming.

The best way to experience these learning objectives would be to connect the VMs of two students together and communicate between your machines. However, setting that up would require a static IP address and open ports from your ISP. I suspect that many of you do not have this enabled at home.

The sample code in directory sample_src is an example of a networked TCP echo server and associated client. A copy of this code is in the labwork_src directory for you to modify.

Once started, the server opens a listening/passive socket and waits for connection requests from a client. This is a very simple server: it accepts at most one connection request. Once a client connects, the server waits for the client’s message. When it receives a message from a client, it simply replies back with the same note, and then the server shuts down.

When the client program is run, it immediately attempts to connect to the server on the specified IP and port given in the command line arguments. The client program then prompts the user and waits for user input. Once the user presses enter, the client sends the message to the server and then waits for a response. Upon receiving the response the client will print the message and terminate.

This sample program illustrates the basics of socket programming about how to create socket, how to listen/accept/connect and how to send/recv to/from the connected host. This is similar to the example shown and discussed in the lecture.

Connecting to the client/server locally in your VM

To run both the client and the server programs in the same VM  we will use a local network connection via a localhost address.

The localhost is the default name describing the local computer address also known as the loopback address. The loopback address is an special IP address used to tell the network adapter to send the packets directly back to the kernel rather than out to the network. For example, typing: ping localhost would ping the local IP address of 127.0.0.1 (the loopback address). When setting up a web server or software on a web server, 127.0.0.1 is used to point the software to the local machine.

Essentially, using the loopback interface bypasses any local network interface hardware. The local loopback mechanism is useful for testing software during development, independently of any networking configurations. For example, if a computer has been configured to provide a website, directing a locally running web browser to http://localhost may display its home page.

Steps to run  the sample code in your VM (locally on a single machine):

  1. Open 2 terminal instances. The multiple terminal windows will allow you to run and view the output of each of the programs simultaneously. One alternative to opening multiple instances of your terminal is to use a tmux session (alternate article) which gives you advanced window options like split screen ($ sudo apt install tmux in your VM). If you choose to use tmux, open up a tmux cheat sheet for the keyboard shortcuts.
  2. Enter the lab directory in $ cd lab7 
  3. Build the programs using the makefile $ make sample
  4. In the first terminal window, run the server: $ ./bin/server_sample -p 20001 
  1. 20001 is a random port number we selected. You may select any port number in the registered port range to use. If you do, then you must use this port number in all the commands going forward.

Port numbers can range from 0 through 65535. The well-known ports (http, ssh, ftp, etc) use the range from 0 through 1023. Registered ports are temporary ports, usually used by clients, and will vary each time a service is used. Registered ports are also called ephemeral ports, because they last for only a brief time. The port is then abandoned and can be used by other services. The registered ports are numbered from 1024 through 49151. Certain port numbers - the well-known port numbers and the registered port numbers - are registered and administered by the Internet Corporation for Assigned Names and Numbers (ICANN) for use by certain classes of applications. The remaining ports, referred to as dynamic ports or private ports, are numbered from 49152 through 65535.

As you are working locally in your VM, you can't conflict with other students and their running servers. However, if everyone was using the same machine, openlab for example, you could conflict with each other or connect to each other's servers.

  1. In the second terminal window, run the client: $ ./bin/client_sample -h 127.0.0.1 -p 20001 
  2. Test your connection between the client and server by typing anything in the client window and pressing enter. The server will receive this and "echo" the message back to the client. The server terminates the connection after completing 1 echo.

Once you are comfortable with the base code provided, modify the client/server program to do the following:

  1. The client and server programs should NOT exit after one round of communication. Each client-server connection should loop until the client closes the connection:
  1. the client waits for the user’s input and sends the input to the server when the user presses enter
  2. server waits for the connected client’s message and sends the message back to the client via the socket.
  3. The client, upon receipt of the response from the server, prints out the message and then re-prompts the user for more input.
  1. Modify the client program to close the connection with the server when the user types "exit" for the message. If needed, refer to the shell base code for reference.
  2. Modify the server program to loop back to listen for a new client connection when the client connection is closed, instead of terminating the server program. The server should not terminate unless Ctrl-C or SIGKILL is received. (It is a long running program)

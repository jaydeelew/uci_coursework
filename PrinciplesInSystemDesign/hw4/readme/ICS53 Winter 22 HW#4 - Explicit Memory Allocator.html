The Allocator

In this assignment, the goal is to create an explicit free list allocator with first-fit placement, block-size ordered, coalescing, and block splitting without splinters.  Specifically, implement a version of the malloc and free functions.

Explicit Free List Management Policy

You MUST use an explicit freelist, as described in Chapter 9.9.13 Page 862, to manage free blocks. Your allocator will manage your freelist by inserting newly freed blocks in the list via block-size order (smallest to largest). In the case of same sized blocks, the newly freed block is inserted into the freelist in front (closer to the head of the list) of the existing blocks.  

Block Placement Policy

Your allocator will support the first-fit placement policy, as described in Chapter 9.9.7 Page 849.  The first block found which satisfies the request is used. If the entire freelist is searched and no block is found, a request is made for more space on the heap.

Immediate Coalescing

Your implementation performs immediate coalescing upon free() as described in Chapter 9.9.10 Page 850.

Each block uses boundary tags as described in Chapter 9.9.11 Page 851. In Chapter 9.9.11 Page 852 Figure 9.39, the boundary tag (footer) is defined as a single 32-bit double word with the block size and allocated bit. Your boundary tag must be a single memory row of 64-bits in a similar format.

Splitting Blocks & Splinters

If possible, the allocator upon malloc allocation must split off  usable excess space into a new free block. It may be possible to split the block into two blocks (one for the requested allocation, one new smaller free block) if the selected free block is larger than the requested allocation. This will help reduce the amount of internal fragmentation. Details about this feature can be found in Chapter 9.9.8 Page 849.

When splitting blocks your allocator MUST NOT create splinters. Splinters are small (< 32 bytes, in this case) groups of free bytes left after inserting a relatively large payload into free space. To avoid this your allocator must “over” allocate the amount of memory requested by the malloc call so that these small useless blocks are not inserted into the free list, but remain with the allocated block.

Blocks

In Chapter 9.9.6 Page 847 Figure 9.35, the header is defined as a single 32-bit value with block size and the allocated bit. In this assignment, the header is defined as a single 64-bit value, with a layout similar to the header in the textbook.

We will define the term Memory Row to represent 64-bits in size.

Block Header Format: 

+-----------------+-------------------+----------------+
|  requested_size |        hid        |   block_size   |
|    (in bytes)   |  (0xAAAAAAAA)     |   (in bytes)   |
|      16 bits    |      32 bits      |    16 bits     |
+-----------------+-------------------+----------------+

+-----------------+-------------------+------------+---+
|  requested_size |        hid        |            | a |
|    (in bytes)   |  (0xAAAAAAAA)     |            |   |
|     16 bits     |      32 bits      | 15 bits    | 1 |
+-----------------+-------------------+--------+---+---+

Block Footer Format: 

+-----------------+-------------------+----------------+
|  requested_size |        fid        |   block_size   |
|    (in bytes)   |   (0xDDDDDDDD)    |   (in bytes)   |
|      16 bits    |      32 bits      |    16 bits     |
+-----------------+-------------------+----------------+

+-----------------+-------------------+------------+---+
|  requested_size |        fid        |            | a |
|    (in bytes)   |   (0xDDDDDDDD)    |            |   |
|     16 bits     |      32 bits      | 15 bits    | 1 |
+-----------------+-------------------+--------+---+---+

  • We will restrict the maximum size of memory managed by the allocator to ~216. Therefore the block_size can never exceed this value. The block_size field represents the total number of bytes for the entire block (header, payload, padding - when needed, footer). The least-significant bit of the block_size field is used as the allocated bit (as explained in the textbook).
  • Within the block_size field of the header and footer the least significant bit is used as a flag. The least significant bit, position 20 labeled a,  denotes if the block is allocated (1)  or not (0). This bit must be extracted from the block_size field manually. There is no field in the struct.  
  • The requested_size field represents the number of bytes requested in the malloc call.
  • The hid and fid fields represent bits that are used by your allocator to identify if a header/footer is valid. The hid field MUST BE SET in the header to 0xAAAAAAAA and the fid MUST BE SET to 0xDDDDDDDD in the footer. Macro constants for these values are defined in the header file for you to use (the notation UL at the end of the value denotes Unsigned Long).

As the largest primitive data type in the system is 16 bytes in size, the payload of each block must be 16-byte aligned. Why? Since the allocator does not know what data you plan to store in the dynamically allocated space, it must ensure that the caller can store any primitive data type without alignment error.

As such, the following constraints apply to all free and allocated blocks created by the allocator:

  • Each block begins with the header (8 bytes), which must have an address that is 8-byte aligned, but not 16-byte aligned.
  • The minimum payload size is 16 bytes (for next and prev pointers for explicit list).
  • Each block ends with a boundary tag  (8 bytes), aka footer, which must have an address that is 16-byte aligned.

Therefore, the minimum block size is 32 bytes (header + payload + footer). The block size will always be a multiple of 16 bytes due to alignment.

ics_malloc

/*
 * This is your implementation of malloc. It acquires uninitialized memory from  

 * ics_inc_brk() that is 16-byte aligned, as needed.
 *
 * @param size The number of bytes requested to be allocated.
 *
 * @return If successful, the pointer to a valid region of memory of at least the
 * requested size is returned. Otherwise, NULL is returned and errno is set to
 * ENOMEM - representing failure to allocate space for the request.
 *

 * If size is 0, then NULL is returned and errno is set to EINVAL - representing

 * an invalid request.
 */
void *ics_malloc(size_t size);

The allocator will start with no space allocated on the heap. When the user program makes the first call to ics_malloc, the allocator must request space to manage using ics_inc_brk(). Multiple calls might be needed in order to satisfy the size of the first malloc request. The allocator then initializes the entire returned space into a single free block for the allocator to manage. The freelist_head pointer points at this block.  A prologue and epilogue will be needed to ensure proper alignment!

Note: We are not using the special prologue block which is mentioned in the textbook for the invariant form on page 855-856 and Figure 9.42. Your prologue (and epilogue, for that matter) should only be 8 bytes (with size of 0 and allocated bit set).

The prologue is an ics_footer and the epilogue is set as an ics_header. Both are set with  requested_size and block_size of 0 and allocated bit set.

Once the allocator has memory space to manage, then the first and all subsequent calls to ics_malloc can be handled identically. ics_malloc selects the appropriate block in the free list according to the block placement policy (first-fit, start searching at freelist_head). If the selected block is larger than required and would not not produce a splinter block when split, then the block is split according to the split policy. Set the headers and footers for the selected block and (if necessary) the split block.  Place the newly created split block into the free list according to the block placement policy (block-size ordered).Return the address of the payload in the selected allocated block.

If the entire free list is traversed and a free block was not selected, more memory space must be requested from ics_inc_brk.

  • If no additional heap space is available, return NULL and  ENOMEM as prior specified. Remember, a maximum of 5 pages of memory can be requested from ics_inc_brk.  
  • If additional heap space is needed from ics_inc_brk, request 1 page at a time. Coalesce this new memory space with the free block at the end of the previously managed heap space (if applicable) or create a new block and place it in the free list according to the block placement policy (block-order). Try again to satisfy the malloc request. (Note this step should be repeated, until ics_inc_brk fails.)  

If the last block of the list is allocated, freelist_head is set to NULL.

ics_free

/*
 * Marks a dynamically allocated block as no longer in use and coalesces with

 * adjacent free blocks. Adds the block to the free list according to the block

 * placement policy.

 *
 * @param ptr Address of dynamically allocated memory returned by the function

 * ics_malloc.

 *

 * @return 0 upon success, -1 if error and set errno accordingly.

 *

 * If the address of the memory being freed is not valid, this function sets errno

 * to EINVAL. To determine if a ptr is not valid, (i) the ptr is in the managed

 * heap space, (ii) check the hid field of the ptr's header for  0xAAAAAAAAAA

 * (iii) check the fid field of the ptr's footer for 0xDDDDDDDDDD, (iv) check

 * that the block_size in the ptr's header and footer are equal, and (v) the allocated bit is

 * set in both ptr's header and footer.
 */
int ics_free(void *ptr);

When implementing your ics_free function, consider the case where an invalid pointer is passed to the function. Before freeing this pointer, your ics_free function should check if the pointer being freed actually belongs to an allocated block.

How? This is not 100% fool proof, but you can do your best to try and ensure this by performing checks on what should be the header of the block. At a minimum you should be checking the following:

  • ptr is in between the prologue and epilogue
  • The allocated bit a is set in both the blocks's header and footer  
  • Check the hid field of the block's header for 0xAAAAAAAA (defined in icsmm.h)
  • Check the fid field of the block's footer for 0xDDDDDDDD (defined in icsmm.h)
  • Check that the block_size in the block's header and footer are equal
  • Check that the requested_size in the block's header and footer are equal

If this pointer does belong to an allocated block, based on the above checks, then the block is to be inserted into the free list. In this case, coalescing may be required. Check the header and footer of the adjacent blocks . Remove the adjacent free blocks from the linked list, coalesce with the newly free'd block (updating the appropriate headers and footers), and insert the newly formed block back into the free list. When successful (with or without coalescing), ics_free returns 0. Otherwise, ics_free returns -1 and sets errno to EINVAL.

Note, the requested_size field is not used when a block is unallocated and is not modified on free. The field will be overwritten when the block is next allocated (which means on allocation, always set requested_size).

Block Headers & Footer

We have repeatedly studied and utilized structures for casting of memory space this term. All that work was to prepare for this assignment!

The block header and footer formats and field sizes are specified in include/icsmm.h.

A block header struct is defined for you as:

#define HEADER_MAGIC 0xAAAAAAAAUL

#define FOOTER_MAGIC 0xDDDDDDDDUL

#define REQUEST_SIZE_BITS 16

#define HID_SIZE_BITS 32

#define BLOCK_SIZE_BITS 16

#define FID_SIZE_BITS 32

typedef struct __attribute__((__packed__)) {

    uint64_t     block_size : BLOCK_SIZE_BITS;

    uint64_t            hid : HID_SIZE_BITS;

    uint64_t requested_size : REQUEST_SIZE_BITS;

} ics_header;

Remember, the __attribute__((__packed__)) tells gcc to leave out all padding between members of the struct. In this way, the fields are forcibly placed next to each other.

To assist in accessing the next and free pointers of a free block, an additional struct ics_free_header is defined.

typedef struct __attribute__((__packed__)) ics_free_header {
    ics_header header;

/* Note: These next two fields are part of the payload and only used when

 * the block is free. They are not part of header, but we place them here for

 * ease of access.

     */

    struct ics_free_header *next;  
    struct ics_free_header *prev;  
} ics_free_header;

Lastly, a struct for the boundary tag (footer) is also included and defined as:

typedef struct __attribute__((__packed__)) ics_footer {

    uint64_t     block_size : BLOCK_SIZE_BITS;

    uint64_t            fid : FID_SIZE_BITS;

    uint64_t requested_size : REQUEST_SIZE_BITS;

} ics_footer;

Grading

Your homework submission will be tested in the course class environment ONLY. This ensures that your implementation is fairly graded, as there can be variations between system libraries, operating systems, and architectures.

It is the responsibility of the student to ensure that the assignment compiles in the course environment and in Gradescope.  Assignments which do not compile or run in the course environment will be assigned a grade of 0.

50% of your assignment grade will be awarded for passing the basic tests provided in Gradescope with the assignment. The remaining 50% of the assignment grade will be allocated based on additional tests. Students are heavily encouraged to test their programs beyond the basic tests provided in Gradescope.

ICS53 W22 Lab #9 - Cache

 In this lab, you will:

  • Visualize the relationship between C code and different cache structures
  • Examine performance differences between cache structures

The number of cache levels, the size of each cache, the cache structure (direct mapped, fully associative, n-way set associative), and cache write policy can have an impact on the performance of different applications. The load and store access pattern of the program (especially, the stride with respect to cache size) also plays a significant role in the performance.

Before we begin, refer to "Latency Numbers Every Programmer Should Know" and examine the latency differences of different components of your system over the last 30 years. For reference, note:

  • Each blue square represents 100 blacks squares
  • Each green square represents 100 blue squares
  • Each red square represents 100 green squares

The graphic visualizes the difference in latency between the fastest memory component, L1 cache and cross the globe network traffic. A few high-level observations:

  • Between 1990-2005, major advancements in data compression speed occurred. However, since 2005, no progress has been made.
  • In the last 15 years the advancements have been in mainly hard disk performance and local network speeds, not at the system level (RAM, cache, etc - black square)

Take a minute to reflect on the relative latency between different aspects of your computer system and why optimizing your code to work with the cache and minimize disk/network access is an important task. (The concept of a web cache also plays an important role in your web browser's performance. Can you see why?)

In this lab, the goal is to visualize the difference between the different cache organizations with different access patterns created by simple C loops. To provide the visualization we will use ParaCache, a set of web-based visualizers.

To begin, download lab9.tar and uncompress into your course directory. In each section of the lab, you will be asked to enter the output of the program into the ParaCache simulator/visualizer and answer the question(s) in the Canvas quiz.

In this lab we will be examining cache behavior for the access (load & store) of items in a 2D array. Each time an entry of the array is accessed the lowest 12 bits of the memory address are printed by the program. These partial addresses are for entry in the ParaCache simulator to visualize the cache requests. The main program performs the following:

  1. Creates a 2D array in memory based on N
  2. The array is then filled with random values using row-major order access using fillArray.
  3. The sum of all the random values in the array are accessed in column-major order using sumA.
  4. The sum of all the random values in the array are accessed 4 elements at a time (square formation) via column-major order using sumB.  

Direct Mapped Cache - Basic Analysis

Load the ParaCache direct mapped cache visualizer.

Set the parameters of the simulator and press the Submit button. This will initialize the memory, instructions format and the cache table.

  • Write Policy: Write Back, Write on Allocate
  • Cache Size: 64 (28 bytes total)
  • Memory Size: 4096 (212 bytes)
  • Offset Bits: 4 (24 bytes per cache block)

Examine the size of the Memory Blocks (the width of a row is the size of a cache block), the size of the different fields (tag, index, offset) of the instruction/address breakdown, and the number of rows in the cache table. In the Information section, the calculations for the size of each of the fields is shown.

Compile and run 2dArray.c with the following define statements

#define N 10

typedef int array_t[N][N];

  • Don't worry about the gcc warnings - we are manipulating the addresses to print them
  • Each time you run the program, the addresses will most likely change. This is expected behavior.

The program will print out 3 groups of addresses to the terminal.  The first group (10 addresses per line) is for the fillArray function (access of the array in row-major order). Select STORE from the drop down menu and delete any value in the hex box. Copy each line, one at a time, into the "List of next 10 instructions" box.  To enter each of these addresses for visualization, press the submit button and the simulator will start to process the next address in the  list.  YOU MUST PRESS SUBMIT (and then Next or Fast-forward) to process each of the instructions individually. To visualize the actions of the request on the cache, press the Next button in the Information section.  The fast-forward button will jump through all the cache steps for a single address.  At the bottom of the information section, statistics of the hit and miss will be displayed along with the hit/miss for each address.

Step through the first row of addresses for fillArray using the simulator.

Answer Question 1 on the Canvas Quiz. 

Notice how on each store request in the first 10 addresses, the block is loaded into the cache, the data is the "stored" and the dirty bit is set to 1.

Note: While the dirty bit should be set to 1 after the above store operations, the simulator does not do this. However on subsequent writes, it will set the dirty bit to 1. This is an error in the simulator.

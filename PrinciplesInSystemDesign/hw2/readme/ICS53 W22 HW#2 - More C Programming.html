What is "unit-testing"?

Unit-testing is a type of software testing where individual units or components are tested. The purpose is to validate that each unit of the software code performs as expected. Unit Testing is done during the development (coding phase) of an application by the developers.

Specifically, we are performing white-box testing in Gradescope with the autograder, where the tests are focused on the functionality of each function.

If you would like to create your own set of unit tests to test your code locally, we suggest criterion and cunit. We have found these easy to use and they have decent documentation.

In addition to the above benefits, unit testing allows for awarding partial credit on this assignment on a per-function basis.  The basecode for this assignment also includes main programs which call the functions as another approach for local testing. This enables verification of the way a function is called (eg. the way arguments are passed, etc) and verifies functionality when the functions are called in conjunction with other code. Make sure to test your functions locally and against the gradescope tests to ensure everything is correctly working in both environments.

NAME

53library - Find books based on search criteria from provided library catalog

SYNOPSIS

53library [-n NUM] [-a | -d] [-o OUTFILE] -I ISBN | -D DATE | -G GENRE | -N NAME | -K KEYWORD

53library -H

(refer to man man description section for more information on how to interpret this usage statement)


DESCRIPTION

The program searches a library catalog file (CSV format) read from STDIN selecting books which meet a search criterion. The results of the search are formatted and printed based on the optional criteria.

The program expects each line of the CSV file to have the following fields (in the specified order):

Book Title,ISBN,Author First Name,Author Last Name,Publish Date,Genre(s)

ISBN is a numerical value.

Publish date is provided in the format: mm/dd/yyyy

Genre is a list of categories separated by a | symbol.

Any Book Title or Author names which contain a comma must be enclosed in double quotes.

The program outputs the results of the search in (TSV format) as

yyyy\tAuthor LastName, AuthorFirst Name\t"Book Title"\tgenre,...

REQUIRED OPTION - Only one option can be specified at a time (unless Extra Credit)

-I ISBN

        Search for the book with the ISBN number specified.

-D DATE

Search for books published on or after DATE. DATE must be specified in mm/dd/yyyy format.

-G GENRE

Search for books with GENRE. GENRE is a single genre type.

-N NAME [1/13 1:02pm The getopts case in the basecode was fixed. Check your version]

Search for books with NAME as either the first or last name of the author. Exact/full field match only. No partial matches. 

-K KEYWORD

Search for books with KEYWORD in the title or author name fields.

-H

        Prints the usage statement to STDOUT. All other arguments are ignored.

OPTIONAL OPTIONS

-n NUM

Returns the top NUM results of the search. By default all search results are returned.


-a | -d

Returns the results in (a)scending order or (d)escending order based on the book ISBN. By default search results are returned in order presented in the library catalog file.

-o OUTFILE (lowercase letter o)

Results are printed into the OUTFILE specified, instead of STDOUT.

EXIT STATUS

The exit code indicates whether 53library was able to successfully perform the requested operation, and if not, what kind of error occurred.

0      Successful termination

1      Unsuccessful execution of the program due to invalid argument, eg. an invalid option specified, too many options specified, invalid argument value/format. The usage statement prints to the standard error stream (STDERR)

2      Unsuccessful execution of the program due to an error with library catalog input

3      Unsuccessful execution of the program - Any error with opening, creating, or accessing OUTFILE.

4      Unsuccessful execution of the program - Any error not specified above.

For the sake of simplicity, you may assume all CSV input files provided to the program have:

  • the correct number of columns on each line
  • all specified field values have valid/proper format (date/number/strings/etc)
  • Genres list will not contain duplicates

Part 2: Basic Generic Linked List

In order to store the different genre field(s) for each book, we will use an instance of the provided generic linked list implementation.

The generic linked list has the following structure:

typedef struct node {

    void* data;          // pointer to the data to be stored

    struct node* next;   // pointer to the next node in the linked list

} node_t;

typedef struct list {

    node_t* head;        // pointer to the node_t at the head of the list

    int length;          // number of items in the list

    int (*comparator)(void*, void*);     // function pointer for in order insertion

    void (*printer)(void*, void*, int);  // function pointer for printing the node data

    void (*deleter)(void*);              // function pointer for deleting any dynamically

                                         // allocated items within the data stored

} list_t;

Review the comments at the top of the linkedlist.c file for more explanation and a visual of how the components are structured together. (Refer to the Appendix for more information about the generic linked list and the provided functions)

We will create an ordered linked list for all the genres associated with a book. Each of the genres for the book will be a node_t in the list.  The void* data field in the node_t will be a string with the genre name. Appendix on Generic Linked List

🤔 Why generic linked list?

Generic linked lists are very useful because they can be used to hold different data.

How do we compare two void*?

With generic linked lists, we can not compare two void* because they can refer to any data type. Without knowing the size of information (based on data type) which is referred to, a proper comparison can not be conducted. In order to compare the data types, we have to specify a comparator to use based on the knowledge of the data type. In our assignment, it is required to have the linked list of keywords appear in alphabetical order. To do this, the generic linked list needs a comparator that performs proper type castings to determine the order in which to evaluate the two void* pointers.

A former TA, Bryan Rocha, created a video that explains more about the basics of the generic linked list code, its usage, and the comparator.  Note: The provided linked list basecode contains additional features not discussed in the video. However, the video's concepts are extendable to these features.  

The following functions of the linked list implementation are provided in src/linkedlist.c. Use the implementation as an example of the operation of pointers and structures within a generic linked list data structure.

* @param print Function pointer to print function specifying how to print the data in the linked

**/
